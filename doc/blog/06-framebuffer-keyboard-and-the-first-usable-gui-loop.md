# Framebuffer、键盘和第一个真正可用的 GUI 闭环

在只有串口的时候，一个系统模拟器已经能做很多事了。但真正让它开始有“整机感”的，通常不是更多的 CPU 指令，而是图形和输入。因为从那一刻起，你不再只是看一条串口日志，而是在看一个完整的软件环境是不是能和人交互。

这个项目里，图形路径的第一步非常克制：
- 先做一个基础 framebuffer 设备。
- 后端用 SDL。
- 不追求复杂显卡模型，只追求 Linux/U-Boot 都能认出来、能真正看到画面。

这条路径的好处是实现和验证都很直接。SDL 作为后端足够轻，跨平台也足够实用；而 simple framebuffer 这条设备模型对 Linux 也足够友好，不需要一开始就掉进复杂 GPU 驱动模型里。

不过，图形一旦出来，新问题马上就来了。最初版本虽然能显示，但刷屏非常慢；后来 Linux 日志虽然能进 framebuffer，控制台和键盘路径却并不自然，甚至一度只是把按键转发到 UART，表面上“能输”，实际上并没有形成真正的图形控制台闭环。

最终把这件事做顺，关键是承认一个事实：如果目标是 Linux 上真正的图形控制台，那么就应该给它一个它熟悉的输入设备，而不是想办法把 UART 假装成键盘。于是后面又补上了更正统的键盘路径，例如 PS/2 方向的实现，并让 Linux 自己去把 framebuffer 和输入设备接成 tty。

这类工作看上去不像 CPU 指令那样“硬核”，但它对模拟器成熟度的提升非常大。原因有两个：

第一，它显著提升了人工调试效率。很多问题在串口里看是抽象的，在 GUI 里一下就具象了，比如：
- U-Boot 的界面是不是稳定。
- framebuffer 刷新是不是卡顿。
- 控制台是不是切到了正确的 tty。
- 键盘事件到底有没有被内核真正接收到。

第二，它逼着板级模型更完整。你不能只把 SDL 窗口开出来，还得保证：
- 设备树把 framebuffer 和键盘描述清楚。
- Linux 内核配置愿意启用这条路径。
- 默认控制台参数不会把输出继续留在串口上。

从工程角度看，这个阶段很有代表性。它告诉我们，模拟器的“可用性”不只是 CPU 正确率的函数。只要图形和输入闭环真正打通，很多之前只能靠脚本做的验证，都会突然变成可以被人直观确认的事情。这对后续继续往更复杂的软件栈推进，是非常值得的一步。
