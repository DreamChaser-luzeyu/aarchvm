# 从一个最小 AArch64 解释器走到 Linux 冒烟启动

这个项目最开始的目标并不是“做一个体系结构教科书”，而是做一个能持续往 Linux 启动推进、同时又把性能放在首位的 AArch64 全系统模拟器。这个出发点很重要，因为它直接决定了实现顺序：优先补程序真的能感知到的行为，先让软件链路闭环，再去谈架构上更细的精确性。

最初的核心非常简单：
- 一个纯解释执行的 CPU 核心。
- 一块连续 RAM。
- 极小设备集：UART、timer、中断控制器。
- 一套足够让裸机测试跑起来的寄存器模型和指令子集。

如果只看“能不能执行几条指令”，这件事并不难。真正难的是一旦目标变成 U-Boot、Linux、BusyBox，这个“最小集”会被软件一层层抬高标准。U-Boot 逼你把系统寄存器、异常返回和设备树传递路径做对；Linux 会继续逼你把页表、TLB、异常综合征、timer sysreg、GICv3、权限检查、initramfs、用户态异常都补齐；BusyBox 再往前一步，会逼着你面对 FP/AdvSIMD、控制台输入、shell 交互和更真实的用户态指令覆盖。

这一路走下来，一个很关键的经验是：不要用“最小实现”这个词自我安慰。真正有价值的问题只有一个：当前这份实现，对上层软件来说是不是已经足够真实？如果不是，就继续补；如果是，再去想怎么让它更快。

因此，这个模拟器的成长路径不是按 ARM 手册目录来的，而是按软件栈的压力来的：
- 先有通用寄存器、PSTATE、基本算术和访存。
- 再有同步异常、`ESR_EL1/FAR_EL1/ELR_EL1/SPSR_EL1`。
- 再有 GICv3 和 timer sysreg 的最小闭环。
- 再把 MMU 扩到能支撑内核早期页表。
- 再往上把 Linux 的用户态需求慢慢吃下来。

当项目走到 Linux + BusyBox shell 真能起来之后，整个事情的性质就变了。它不再只是“我能解释一些 A64 指令”，而是“我已经拥有一条真实软件栈会持续施压、持续揭露缺口的反馈回路”。这条反馈回路比任何静态清单都更有价值，因为很多 bug 根本不会在孤立指令测试里出现，只会在 U-Boot、Linux、用户态程序和回归脚本的组合里暴露出来。

所以，如果要用一句话概括这第一阶段，我会说：这不是从零写了一个模拟器，而是从零搭起了一条能够不断逼近真实系统行为的工程链路。
