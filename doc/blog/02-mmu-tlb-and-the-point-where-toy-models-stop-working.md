# MMU、TLB，以及玩具模型为什么会很快失效

在能跑 Linux 之前，很多 AArch64 模拟器实现都会经历一个阶段：MMU 有了，页表也能走，TLB 也能缓存，看起来什么都有了。但只要真的把内核放上去，这类“看起来有”的实现很快就会露馅。

原因很简单。对裸机 demo 来说，地址翻译常常只是“VA 变 PA”。但对 Linux 来说，页表项和表项继承关系里真正关键的是程序可感知语义：
- 哪个映射对 EL0 可见，哪个只允许 EL1。
- 哪个页可写，哪个只读。
- 哪个取指允许，哪个会因为 PXN/UXN 触发异常。
- 换 `TTBR` 时 ASID 到底怎么参与 TLB 命中。
- `TLBI` 指令到底该影响哪一类条目，而不是一把梭清空全部缓存。

这也是为什么项目后期 MMU/TLB 的重点不再只是“走通页表”，而是补这些更细的行为：
- 页表和上级 table attribute 的继承。
- `AP[2]`、`AP[1]`、`APTable` 的组合语义。
- `PAN` 对 EL1 数据访问用户映射的限制。
- TLB 项里显式记录 `asid`、`user_accessible`、权限和执行属性。
- `TLBI ASIDE1/ASIDE1IS`、`VAE1/VALE1`、`VAAE1/VAALE1` 的作用域差异。

这些东西在实现上都不算“宏大工程”，但每一项都很难绕开。因为一旦做错，上层软件通常不是直接崩，而是出现更难查的“偶发正确”：
- 某些用户态页偶尔能访问，偶尔不行。
- 某次 `TTBR` 切换之后旧条目还在命中。
- 某些 `TLBI` 明明执行了，软件却还看见旧映射。

为了避免这种“看起来偶尔对”的状态，最有效的办法不是靠人工推理，而是把行为拆成裸机单测。这个项目后来补的几组测试就非常典型：
- EL0 对 privileged-only 映射的权限 fault。
- `PAN` 阻止 EL1 访问用户映射。
- 不同 ASID 下的 TLB 命中/失效范围。
- `TTBR` 切换后显式携带 ASID 的可见行为。

一旦这些测试稳定通过，MMU/TLB 就从“足够 demo”跨到了“足够支撑内核启动路径”。这中间的区别不是页表代码多了多少行，而是你终于开始模拟 Linux 真正在乎的那部分规则了。

这也是我对“最小实现”这个词的一个修正：最小，不是字段最少、代码最短，而是对目标软件来说不再缺关键语义。
