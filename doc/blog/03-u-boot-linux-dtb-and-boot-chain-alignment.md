# U-Boot、Linux、DTB，以及把启动链路对齐这件事

很多人第一次做全系统模拟时，最容易低估的不是 CPU，而是启动链路的“对齐”工作。内核镜像在哪里、DTB 放在哪里、U-Boot 配置里默认假设的地址是什么、模拟器里的 RAM 基址是不是和这些配置一致，这些细节一旦不一致，就会产生一种非常折磨人的状态：每一层看起来都“差不多对”，但系统就是起不来。

这个项目里有过一个很典型的阶段：U-Boot 能执行，也能打印，但一旦往 Linux 推进，就会不断撞上“设备树无效”“地址不匹配”“参数传递路径有问题”这类错误。最后真正把问题理顺，靠的不是某一个大修，而是一系列很具体的对齐工作：
- 把 RAM 基址统一到 `0x40000000`，不再继续把 `0x0` 当作内存基址。
- 让 DTB 放在独立 RAM 区间，而不是偷偷重映射到 `0x00000000`。
- 保持 U-Boot 本体加载位置稳定，同时正确通过 `x0` 传递 DTB 指针。
- 按当前板级配置重新审视 U-Boot 里与 kernel、initrd、fdt 相关的默认地址。

这里最值得强调的一点是：设备树不是“有就行”的装饰品。对于 Linux 来说，它几乎就是早期世界模型的一部分。UART、GIC、timer、framebuffer、keyboard，这些设备到底存在不存在、在哪个 MMIO 地址、irq 号是多少、compatible 是什么，都要靠 DTB 来把内核和模拟器对上。

而一旦板级模型往前演进，比如后来加了简单 framebuffer 和 PS/2 键盘，这种对齐工作还得继续做。你不能只在模拟器里把设备实现出来，还得确保：
- 设备树把它们描述出来。
- U-Boot 或 Linux 的驱动路径愿意接管它们。
- 命令行和控制台配置不会把输出重新拉回串口。

这也是为什么“先让设备看起来工作”远远不够。启动链路真正的难点在于，模拟器、U-Boot、设备树、内核配置必须形成一致的叙述。任何一处还在说旧故事，系统都会在最意想不到的地方掉链子。

当这件事做对以后，收益非常大：不只是 Linux 能往前走，后面的所有调试也都会轻松很多。因为从那一刻起，你看到的内核日志和设备行为才真正开始指向同一个系统，而不是几个彼此错位的半真半假世界。
