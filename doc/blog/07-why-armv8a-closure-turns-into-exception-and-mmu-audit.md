# 为什么 Armv8-A 的“收口”最后会变成异常与 MMU 审计

当一个 AArch64 模拟器刚开始能跑起来时，最显眼的工作总是“还差哪些指令”。但一旦它已经能启动 Linux、能跑用户态程序、还能支撑一批裸机样例，问题的重心就会变掉。

这时候最难的部分，往往已经不是“有没有解码到某条 opcode”，而是：
- 某条本来已经能跑的指令，在异常路径上是不是仍然符合架构要求。
- 快路径、TLB、预解码这些为了性能引入的缓存，是否和慢路径保持一致。
- 程序真正能观察到的寄存器副作用，尤其是 `ESR_EL1`、`FAR_EL1`、`PAR_EL1`、`ELR_EL1`，是不是在边界条件下也对。

这也是这段时间反复收口之后，我最明确的体会之一：Armv8-A 的后期收口，本质上是在做“程序可见行为的总对账”。

## 指令“存在”不等于行为“闭合”

很多 bug 都不是因为某条指令根本没实现，而是因为它在某个细分语义上没有和周边系统闭合。

一个典型例子，是系统寄存器写入和缓存状态的关系。比如：
- 写 `MAIR_EL1` 后，旧的 TLB 项如果仍缓存着解析好的 memory attribute，那么后续访问即使经过 `ISB`，程序仍可能观察到旧属性。
- 切换 `SCTLR_EL1.M` 开关时，如果旧的 TLB 结果没有被正确清掉，那么“MMU 关闭后 VA 应直接等于 PA”的语义就会被 stale entry 污染。

这两类问题，从“指令覆盖率”角度看都很隐蔽，因为：
- `MSR` 本身已经实现了。
- 访存也已经能成功。
- Linux 甚至可能在大多数启动路径里看不出问题。

但从程序可见行为角度看，它们是实打实的错。因为 guest 看见的是错误的地址翻译结果或错误的内存属性，而不是“实现细节不同”。

## 真正难的是异常侧的对账

再往后走，问题会进一步集中到异常。

这是因为很多指令在“正常执行”路径上并不复杂，真正复杂的是它们失败时 guest 应该看见什么：
- `EC` 要不要变。
- `ISS` 里哪些位该填，哪些位必须清零。
- `FAR_EL1` 这次到底是否有效。
- `PAR_EL1` 在 `AT` 成功和失败两种情况下格式是否都正确。
- `ELR_EL1` 指向 faulting instruction 还是 preferred return address。

只要这些细节有一处错，轻则某些测试不稳定，重则 Linux 在罕见路径上直接走偏，而且这种问题通常不会在“能不能启动”这一级别上立刻暴露。

所以，收口阶段的主线自然会从“补指令”转成“逐类审计异常家族”。这不是工作方向漂移，而是成熟阶段的必然结果。

## 为什么 fast path 会让这件事更难

如果只是一个纯慢路径解释器，很多正确性问题虽然仍然存在，但审计范围会小很多。可一旦为了性能引入：
- decode cache
- predecode
- bus fast path
- TLB 热路径

那么每个行为都不再只有一个实现点，而是变成：
- 慢路径是否正确
- 快路径是否和慢路径一致
- 失效机制是否足够完整

这会直接改变收口方法。后期很多测试其实不是在验证“有没有实现某条指令”，而是在验证“切换状态之后，快路径有没有继续偷偷沿用旧结论”。

这也是为什么后期新增的很多样例，看起来都很“小题大做”：
- 写一个系统寄存器。
- 打一个 `ISB`。
- 访问同一个 VA。
- 再切回来重试。

表面上是一个很短的裸机程序，实际上它在钉死的是快慢路径一致性，而这类问题恰恰最容易在系统长期运行里演变成偶发崩溃。

## 收口后期，测试比“补码表”更重要

到了这个阶段，我越来越不相信“我读过这段代码，所以它应该是对的”。更可靠的做法是：
- 先找出程序可见的合同。
- 再构造一个能最短路径观测这个合同的样例。
- 最后把它放进正式回归，而不是停留在一次性验证。

这也是近期很多工作看上去像是在“补测试”，其实本质上是在补证明链。因为没有一条可重复执行的样例，很多收口结论只是印象，不是证据。

## 结语

如果说前期的任务是把模拟器从“不能跑”推进到“能跑”，那么后期的任务就是把它从“经常看起来没问题”推进到“程序看到的行为真的闭合”。

而在 Armv8-A 这里，最后这一段路，几乎一定会落到异常、MMU、TLB、fault、快慢路径一致性上。不是因为别的地方不重要，而是因为只要系统已经能跑起来，剩下最难、也最值得收口的，通常就只剩这些了。
