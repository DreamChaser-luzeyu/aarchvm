# 反复收口之后，我对 Armv8-A 模拟器有了哪些新认识

近期这段时间，围绕 Armv8-A 最小程序可见集合，做了很多轮“审阅代码、补测试、跑全回归、再回来看还有没有漏”的工作。过程很重复，但收获其实非常具体。

如果要把这些经验压缩成几条，我觉得最重要的是下面这些。

## 1. “能启动 Linux”只是开始，不是结论

这是最容易误判的一点。

一个 system emulator 只要能把 Linux 启动到 shell，人很容易产生一种错觉：CPU 大体已经差不多了。可后面一轮轮问题证明，这个判断经常过于乐观。

因为 Linux 启动成功只能说明：
- 主路径大致能走通。
- 一批常见指令和异常至少没错到立即崩掉。

但它说明不了这些事：
- 稀有 fault 的 syndrome 是否正确。
- `AT` / `PAR_EL1` / `FAR_EL1` 是否在边界上对。
- 快路径缓存是否会在状态切换后串味。
- 某些只在长期运行、SMP、重度刷屏、反复缺页时才出现的错有没有被藏住。

所以后来我的判断标准越来越保守：Linux 能跑，只能说明“值得继续审”，绝不是“可以宣布完成”。

## 2. 后期很多真实 bug 都长得不像“缺指令”

前期最典型的问题是：这条没实现，那条解码错了。后期不是。

后期更常见的是：
- 某条系统寄存器写了，但相关缓存没失效。
- 某类异常进去了，但 `ISS` 还差几位。
- 某条访存慢路径是对的，快路径却在边界上漏了一个条件。
- 某次地址翻译本来该失效，却因为旧 TLB 项还在而得到“貌似合理”的结果。

这些 bug 的共同点是：它们都不是“完全不会工作”，而是“只在某个很具体的边界上不闭合”。这也是为什么后期收口的感觉更像审计，而不是开发。

## 3. 最有价值的测试，通常不是最大的测试

我以前也会下意识觉得，既然问题在 Linux 里暴露，就应该用 Linux 去验证它。但后来发现，最有价值的样例通常反而非常小。

原因很简单：
- 大系统能告诉你“哪里可能有问题”。
- 小样例才能告诉你“到底是哪条语义没闭合”。

因此比较好的工作方式通常是：
- 用 Linux 或复杂 workload 暴露症状。
- 用最小裸机样例锁定合同。
- 用完整回归验证修复没有外溢破坏。

这三层不是互相替代，而是各司其职。

## 4. 高性能优化和正确性收口，后期已经分不开了

这也是这段时间最大的认识变化之一。

一开始会觉得：
- 性能优化是性能优化。
- ISA 收口是 ISA 收口。

但做着做着就发现，在有 predecode、decode cache、TLB fast path、bus hot path 的实现里，两者早就耦合在一起了。

因为任何一个缓存层都会引入两个问题：
- 它本身是不是算对了。
- 它什么时候必须失效。

所以后期很多“正确性 bug”其实直接来自性能优化后的状态缓存；而很多“性能设计是否合理”的判断，也离不开对架构边界的理解。

换句话说，高性能解释器的后期工程，不存在纯粹的“性能话题”或“正确性话题”，只有“在不破坏程序可见语义的前提下，怎样把热路径做短”这个联合问题。

## 5. 真正决定收口速度的，不是写代码速度，而是排除错误方向的速度

这类工作最耗时间的，往往不是改一行代码，而是确认哪些方向不该走。

比如：
- 这是不是一个外设问题，其实根因在 CPU？
- 这是不是串口/SDL 路径问题，其实是异常返回或原子语义问题？
- 这是不是某条指令没实现，其实是 TLB / predecode 失效没跟上？

每次能更快排除一条错误方向，收口就会快很多。反过来，如果一开始问题定义就错了，后面再勤奋也只是绕圈。

所以我现在越来越重视“先把问题翻译成一个精确的程序可见断言”。这件事虽然看起来不像编码，但往往决定了后面到底是一下午修完，还是又多绕三轮。

## 6. 什么时候才能说“真的收口了”

这可能是最难回答的问题。

我的看法是，至少要同时满足三件事：
- 已实现功能范围内，高风险异常家族的 `ESR/FAR/PAR/ISS` 已逐类对账。
- `MMU/TLB/fault` 与 fast-path / predecode 的关键边界已经被正式回归压住。
- Linux 和用户态压力路径不再反复暴露新的 CPU 级行为 bug。

只满足前两条而没有第三条，说明实现可能还没经受够系统压力；只满足第三条而没有前两条，则说明只是“暂时没撞到”。

也就是说，真正的收口不是一句“我觉得差不多”，而是代码审计、最小样例、系统回归这三条证据链同时收敛。

## 结语

如果以后再让我从头做一遍，我会更早把这件事当作“长期审计工程”，而不是“补完一份指令清单”。因为 Armv8-A 到后期真正难的，从来都不是表面上的 opcode 数量，而是整套程序可见行为有没有被一个高性能实现稳定、可重复地守住。

而这恰恰也是这段时间反复收口之后，最值得留下来的经验。
