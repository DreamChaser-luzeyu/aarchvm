# 我是如何设计这套 Armv8-A 收口工作流的

反复做了很多轮 Armv8-A 行为收口之后，我越来越确定：如果没有一套严格工作流，后期很容易陷入“今天修一个角落，明天又在别处回退”的循环。

这个项目后来逐渐形成的工作方式，并不是一开始就完整想好的，而是在一次次踩坑后被逼出来的。它的核心目标只有两个：
- 保证修改是真修，不是绕。
- 让每次收口都能沉淀成下一次不会再退化的证据。

## 这套工作流为什么必须存在

CPU 行为类 bug 和普通业务代码 bug 不太一样。它有几个非常麻烦的特点：
- 很多错误只在边界条件下出现，平时 Linux 也能启动。
- 一旦为了性能引入快路径，同一语义往往有不止一个实现点。
- 很多问题会以“随机 panic”“偶发乱码”“某个命令偶尔 SIGILL”这种表象出现，根因却藏在更底层。

如果没有流程约束，人会很自然地滑向几种错误做法：
- 靠工作负载特判绕过。
- 只看 Linux 能不能启动，不验证寄存器副作用。
- 发现一个 bug 后直接改代码，但不补最小样例。
- 回归只跑一遍，而且只跑最容易过的路径。

这些做法短期很省事，长期代价极高。因为你会失去对“到底修好了什么”的控制。

## 我最终采用的流程

后面逐渐固定下来的，是一个五段式循环。

### 1. 先把问题翻译成“程序可见合同”

不要一上来盯着实现细节，而要先回答：
- guest 到底应该观察到什么？
- 哪个寄存器、异常码、返回地址、内存结果是可观察的？
- 哪种行为是架构明确要求的，哪种只是内部实现自由度？

这个步骤非常关键，因为它会直接阻止很多“看上去合理”的错误修法。只要合同写不清，后面的改动很容易变成拍脑袋。

### 2. 然后找最短观测路径

一旦合同明确，下一步不是先改代码，而是先想办法构造一个最短样例去观测它。

我后面越来越偏爱裸机最小样例，原因很简单：
- 它短。
- 它可控。
- 它失败时定位半径小。

例如想验证 `MAIR_EL1` 写后旧 TLB 属性是否失效，最好的办法不是去看 Linux 有没有随机崩，而是写一个只做：
- 建页表
- 触发一次翻译填 TLB
- 改 `MAIR_EL1`
- `ISB`
- 重访存

这种样例往往几十行就够了，但它对行为的约束，比长系统日志有力得多。

### 3. 只在证据明确后改实现

当最小样例已经把问题钉死，再去改 CPU 行为。这个顺序很重要，因为它会强制实现服从证据，而不是反过来让证据为实现找借口。

这一阶段我的原则一直比较硬：
- 不做特殊 PC 特判。
- 不做按字节模式识别 workload 的快修。
- 不接受“这个程序现在能跑了所以应该没问题”。

如果改动不能解释为更接近程序可见合同，那就不是合格修复。

### 4. 裸机样例通过后，再跑完整系统回归

很多 CPU 级修复在单测里是对的，但一放进系统里就会暴露连锁问题，所以不能停在裸机层。

我后来基本固定为两层验证：
- 第一层：最小裸机样例，验证语义。
- 第二层：完整 Linux 回归，验证这次修复没有破坏已有系统行为。

而且系统回归不能只跑单核、只跑一条路径。SMP、块设备、图形路径、用户态功能回归，最好都要覆盖到。

### 5. 最后把结论写进文档和待办

这一段看起来像“管理工作”，其实它对后续效率很重要。

因为 CPU 收口这种事情最怕记忆污染。若不把下面这些写清楚：
- 这轮到底修了什么。
- 哪类异常已经对到账什么程度。
- 还剩哪些风险边界没封住。

那么过几天再看时，人会反复回到同一个问题上，或者误以为某件事已经完全做完。

所以我后来会同时维护：
- 正式回归样例
- `TODO.md` 中的收口状态
- 必要的审计文档或修改日志

这三者合在一起，才构成“这轮工作已经落地”的闭环。

## 为什么这套流程适合高性能解释器

这个项目的目标不是做一个微架构级精确模拟器，而是做一个高性能、但程序可见行为正确的 system emulator。这个目标决定了工作流不能照搬“精确硬件模型”的打法。

对我们来说，最重要的是：
- 不强求内部过程和真机一模一样。
- 但必须保证 guest 能观察到的 ISA 行为符合规范。

这意味着很多判断都要围绕“程序可见性”展开，而不是围绕“内部实现长得像不像手册伪代码”展开。也正因为如此，这套流程特别强调：
- 先定义可见合同
- 再做最小观测
- 再验证快慢路径一致性

它天然适合这种高性能导向的实现。

## 这套工作流带来的最大收益

最大的收益不是“每次都能很快修好”，而是“即使修得慢，也越来越不容易退回去”。

收口工作本来就枯燥、重复、局部、需要大量耐心。如果没有方法论，它很容易变成徒劳劳动；但一旦每轮都能留下样例、回归和文档，项目就会开始出现一种很关键的性质：它在稳步变得更难被改坏。

对一个要继续往 SMP、更多外设、更多 guest 软件栈扩展的模拟器来说，这一点比单次修复速度更重要。
