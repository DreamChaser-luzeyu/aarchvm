# 预解码、fast path，以及热点到底在说什么

当一个解释器已经能跑 Linux 之后，性能优化就不再是“看到哪里慢就内联一下”这么简单了。你会很快遇到一个现实：很多看上去很直觉的优化，最后并不一定真的更快。

这个项目后期做过很多轮性能实验，结果非常有代表性。比如：
- 总线 fast path 明确能带来收益，尤其是在 RAM 访问上直接命中时。
- decode cache / predecode 让一部分高频整数路径显著降了成本。
- IRQ 查询门控和 O(1) pending 状态也能去掉不少固定热负担。

但与此同时，也有一些“直觉上应该更快”的尝试被证明确实不值得保留：
- 为数据侧 RAM 再加一层页缓存。
- 为取指侧再加一层最近页快路径。
- 一些只减少一两层 helper 调用的小改动。

这些失败尝试最有价值的地方，不是证明它们“没用”，而是说明热点真正在哪里。后面的 `gprof` 结果反复指向同一组核心路径：
- `Cpu::step()`
- `Cpu::lookup_decoded()`
- `Cpu::translate_address()`
- `Cpu::exec_load_store()`
- `Cpu::exec_data_processing()`
- `BusFastPath::read()` / `Bus::read()`

这意味着解释器现在的主要成本已经非常集中。系统不是“到处都慢”，而是有三条明确的高压链路：
1. 取指之后的 decoded 命中链。
2. 虚拟地址到物理地址的翻译链。
3. 真正执行 load/store 时的访存链。

一旦热点长时间稳定在这三条链路上，优化策略就应该变化。继续做零碎 helper 内联，收益会越来越低；更值得投入的是结构性手段，比如：
- 让更多高频整数路径走专门的 decoded 执行闭环。
- 让 decoded load/store 更直接复用 TLB 和 RAM hot path。
- 把更多解释期开销前移到 decode 阶段，但同时保留自修改代码和地址空间切换下的正确失效机制。

这就是为什么后面项目会开始认真讨论“是否朝 JIT 迈进”。不是因为解释器已经完全没法优化了，而是因为热点已经高度收敛，继续靠局部修枝，很难再拿到数量级收益。

换句话说，热点分析给出的不只是“哪里最慢”，更是在告诉你：下一阶段的优化应该继续做微观打磨，还是该升级整套执行模型了。
