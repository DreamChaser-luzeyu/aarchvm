# 从裸机单测到 Linux 用户态回归：让正确性不靠运气

模拟器开发有一个很现实的问题：如果没有成体系的测试，任何一次“看起来很小”的修改都可能在几天以后变成一次很难定位的回归。尤其是 CPU、MMU、异常和设备交织在一起之后，靠人肉启动 Linux 看两眼日志，几乎不可能保证改动真的安全。

这个项目后期逐渐形成了一套分层测试方法，我认为这是它能稳定推进到 Linux/BusyBox 的关键原因之一。

第一层是裸机单元测试。它的价值在于把一个复杂系统拆回最小可控样本：
- 指令语义是不是对。
- 某个异常是不是按预期进 vector。
- 某个系统寄存器访问是不是回了正确值。
- 某条 `TLBI` 或权限检查是不是只影响该影响的范围。

第二层是 Linux 功能回归。这里已经不是测“某条指令”，而是测“用户态能不能真实做成一件事”：
- `uname -a`、`ps`、`ping`、`df` 这类 BusyBox 命令是否正常。
- initramfs 是否正确挂载并执行 `/init`。
- shell 是否可交互。
- FP/AdvSIMD 与整数路径是否会在用户态真实 workload 中触发 `Illegal instruction`。

第三层是算法/性能回归。这个层次不只是为了快，而是为了稳定地观察热点分布：
- base64、FNV1a 这类 workload 能让纯整数和字符串路径持续受压。
- TLB 顺序/随机访问 workload 能把地址翻译链路单独放大出来。
- 通过统一 initramfs 和 snapshot，性能比较终于不再被完整冷启动成本掩盖。

我很喜欢这个体系的一点是：它并不把“单测”和“系统测试”对立起来。两者的职责非常清楚：
- 裸机单测负责把 bug 精确定位到某个语义点。
- Linux 回归负责保证这份语义真的足够支撑真实软件。

例如，后面补 `CPACR_EL1.FPEN` trap 时，如果只靠 Linux 看，很难快速判断是 EL1 trap 错了、EL0 trap 错了，还是 `FPCR/FPSR` 系统寄存器路径漏掉了。但有了单独的裸机测试，就能先把 EL1 和 EL0 两条路径直接钉死；然后再跑 Linux 功能回归，确认这层语义不会把现有用户态路径打坏。

这类分层测试最大的意义，不是让 CI 看起来漂亮，而是让工程推进不再依赖“运气好这次没坏”。对模拟器这种底层系统来说，这一点几乎等于生产力。
